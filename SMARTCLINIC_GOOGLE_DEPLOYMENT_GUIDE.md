## 1. Overview

SmartClinic uses **Google Calendar API v3** with two tiers of OAuth:

| Tier | Purpose | Credential Source |
|------|---------|-------------------|
| **Global (system)** | Sync all bookings, free/busy queries, watch channels, drift checks | `GOOGLE_REFRESH_TOKEN` in `.env` — a single offline token for the clinic calendar account |
| **Per-doctor** | Each doctor connects their own Google Calendar | OAuth flow via `/api/doctors/connect-calendar` — tokens stored in `DoctorCalendarToken` table |

### Services Used

| Service | API Version | Enabled By |
|---------|-------------|------------|
| Google Calendar API | v3 | `googleapis` npm package |
| Google OAuth 2.0 | — | `google-auth-library` (bundled) |
| Google Meet (Hangouts) | — | Part of Calendar API events |
| Google People API | v1 | `calendarList.list()` during OAuth callback |

### Architecture

```
OAuth Flow:
  User → /api/doctors/connect-calendar?doctorId=X → Google consent screen
       → redirects to /api/google/oauth2callback?code=...&state=...
       → exchanges code for tokens
       → stores in DoctorCalendarToken table
       → calls calendarList.list() to discover primary calendar ID
       → updates doctor.calendarId

Push Notifications:
  watchCalendar() → calls google.events.watch() with webhook URL
  Google sends POST to {NEXT_PUBLIC_APP_URL}/api/google/webhook
  Headers: x-goog-channel-id, x-goog-resource-id, x-goog-resource-state,
           x-goog-channel-token, x-goog-message-number

Sync Flow:
  syncBooking() → creates/updates/deletes Google Calendar event
  On failure → enqueues CalendarSyncJob
  Retry worker (cron every 1 min) → processes queued jobs
  On 410 (expired sync token) → full resync
```

> parameter level, not in the path. This exact pattern `?doctorId=*` is supported.

### Step 6: Generate the Global Refresh Token

The global `GOOGLE_REFRESH_TOKEN` is used by `google.ts:11` for system-level
calendar operations (sync all bookings, free/busy, watch channels, drift checks).

#### Method A: Google OAuth 2.0 Playground (Easiest)

| Step | Action |
|------|--------|
| 1 | Open https://developers.google.com/oauthplayground |
| 2 | Click gear icon (⚙) top right |
| 3 | Check **Use your own OAuth credentials** |
| 4 | Enter your `GOOGLE_CLIENT_ID` and `GOOGLE_CLIENT_SECRET` |
| 5 | In left panel, type/search `https://www.googleapis.com/auth/calendar` |
| 6 | Check the scope → Click **Authorize APIs** |
| 7 | Sign in with the **clinic calendar account** |
| 8 | Click **Allow** on consent screen |
| 9 | Click **Exchange authorization code for tokens** |
| 10 | **Copy the Refresh token** → set as `GOOGLE_REFRESH_TOKEN` |

#### Method B: Direct Authorization URL

Build this URL (replace placeholders with your actual values):

```
https://accounts.google.com/o/oauth2/v2/auth?
  client_id={GOOGLE_CLIENT_ID}&
  redirect_uri={GOOGLE_REDIRECT_URI}&
  response_type=code&
  access_type=offline&
  prompt=consent&
  scope=https://www.googleapis.com/auth/calendar
```

| Step | Action |
|------|--------|
| 1 | Open the URL in a private/incognito browser window |
| 2 | Sign in with the **clinic calendar account** |
| 3 | Click **Allow** on the consent screen |
| 4 | The browser redirects to: `{GOOGLE_REDIRECT_URI}?code={AUTH_CODE}` |
| 5 | Copy the `code` parameter from the URL |
| 6 | Exchange the code for tokens: |

```bash
curl -X POST https://oauth2.googleapis.com/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "client_id={GOOGLE_CLIENT_ID}" \
  -d "client_secret={GOOGLE_CLIENT_SECRET}" \
  -d "code={AUTH_CODE}" \
  -d "redirect_uri={GOOGLE_REDIRECT_URI}" \
  -d "grant_type=authorization_code"
```

Response will contain:
```json
{
  "access_token": "ya29.a0...",
  "expires_in": 3599,
  "refresh_token": "1//0g...",
  "scope": "https://www.googleapis.com/auth/calendar",
  "token_type": "Bearer"
}
```

| Value | Set as |
|-------|--------|
| `refresh_token` | `GOOGLE_REFRESH_TOKEN` |

> **Critical:** The `refresh_token` is only returned **the first time** a user
> authorizes with `access_type=offline` and `prompt=consent`. If you need to
> re-generate, go to https://myaccount.google.com/permissions, find the app,
> and revoke access first, then repeat the flow.

## 4. Cron & Scheduler Configuration (Phase 8)

### 4.1 Retry Worker (Already Configured)

The retry worker endpoint is `POST /api/internal/calendar/retry`.

#### Vercel — Already in `vercel.json`

```json
{
  "crons": [
    {
      "path": "/api/internal/calendar/retry",
      "schedule": "* * * * *"
    }
  ]
}
```

This runs **every minute**. Vercel Cron will automatically call this endpoint
when deployed. No additional Vercel configuration needed.

#### Docker — Already in `docker-compose.yml`

The `cron` service in `docker-compose.yml` runs every minute:
```yaml
services:
  cron:
    image: alpine:3.19
    environment:
      - CALENDAR_INTERNAL_SECRET=${CALENDAR_INTERNAL_SECRET}
    command: >
      sh -c "
        apk add --no-cache curl &&
        echo '*/1 * * * * /usr/bin/curl -s -X POST http://app:3000/api/internal/calendar/retry
          -H \"Authorization: Bearer $${CALENDAR_INTERNAL_SECRET}\"
          -H \"Content-Type: application/json\" --max-time 30' > /etc/crontabs/root &&
        crond -f -l 2
      "
```

#### Auth Mechanism (Two Methods)

The retry worker endpoint (`internal/calendar/retry/route.ts:83-96`) supports:

1. **Bearer token** (preferred): `Authorization: Bearer {CALENDAR_INTERNAL_SECRET}`
2. **Role-based** (fallback): Requires authenticated `superadmin` or `admin` user

The cron infrastructure should use method #1 (Bearer) — both Vercel Cron and
Docker cron do this correctly.

### 4.2 Channel Renewal — NOT YET AUTOMATED

**Endpoint:** `POST /api/cron/renew-channels` (exists as route)

**What it does:** Calls `renewExpiringChannels()` from `channelScheduler.ts`
which finds all channels expiring within 24 hours, stops the old channel, and
creates a new 7-day channel.

**Current status:** There is **no cron schedule** in `vercel.json` or
`docker-compose.yml` for this endpoint. It can only be triggered manually via
the Maintenance tab in the admin dashboard.

#### Recommended: Add to Vercel Cron

Add to `vercel.json`:
```json
{
  "crons": [
    { "path": "/api/cron/renew-channels", "schedule": "0 * * * *" }
  ]
}
```

This runs **hourly**.

#### Recommended: Add to Docker Cron

Add to the `command` in `docker-compose.yml` cron service:
```
0 * * * * /usr/bin/curl -s -X POST http://app:3000/api/cron/renew-channels --max-time 30
```

### 4.3 Drift Check — NOT YET AUTOMATED

**Endpoint:** `POST /api/cron/drift-check`

**What it does:** Calls `runDriftCheck()` from `driftMonitor.ts` which
reconciles all doctor bookings against their Google Calendar events.

**Current status:** There is **no cron schedule** for this endpoint either.

#### Recommended: Add to Vercel Cron

```json
{
  "crons": [
    { "path": "/api/cron/drift-check", "schedule": "0 2 * * *" }
  ]
}
```

This runs **daily at 2:00 AM**.

#### Recommended: Add to Docker Cron

```
0 2 * * * /usr/bin/curl -s -X POST http://app:3000/api/cron/drift-check --max-time 120
```

## 5. Deployment Verification (Phase 9)

### 5.1 Verify Webhook URL

The webhook endpoint must be **publicly accessible** from the internet because
Google's notification servers need to reach it.

| Check | Expected | Validation |
|-------|----------|------------|
| URL | `{NEXT_PUBLIC_APP_URL}/api/google/webhook` | `POST https://your-app.vercel.app/api/google/webhook` returns 200 |
| Method | POST | Any GET request returns 405 or 404 |
| HTTPS | Required | Vercel provides TLS automatically |
| Response | `"OK"` text | Valid webhook returns `new NextResponse('OK', { status: 200 })` |

**For local development:** Google's servers cannot reach `localhost:3000`.
Use a tunneling service:

```bash

ngrok http 3000
# Set NEXT_PUBLIC_APP_URL to the ngrok URL
# Update GOOGLE_REDIRECT_URI to the ngrok URL + /api/google/oauth2callback
# Re-create the OAuth Client ID with the ngrok URL as redirect URI
```

**For production:** Vercel provides HTTPS automatically. The webhook URL
will be `https://your-app.vercel.app/api/google/webhook`.

## 6. Production Verification Checklist (Phase 10)

### 6.1 Calendar Sync

- □ **Booking → Google**: Create a booking → verify event appears in Google Calendar
- □ **Booking update → Google**: Update booking time → verify event is updated
- □ **Booking cancellation → Google**: Cancel booking → verify event is deleted
- □ **Multiple bookings**: Create 10+ bookings → verify all sync correctly
- □ **Resync**: Use the admin dashboard "Resync" button → verify it works
- □ **Recreate event**: Use admin "Recreate Event" → verify old event replaced
- □ **Full resync**: Trigger full resync → verify all events match bookings

### 6.2 OAuth

- □ **Global refresh token**: App starts without auth errors in logs
- □ **Per-doctor connect**: Initiate OAuth via `/api/doctors/connect-calendar?doctorId=X`
- □ **Consent screen**: User sees the correct app name and `auth/calendar` scope
- □ **Token stored**: `DoctorCalendarToken` table has the new tokens
- □ **Calendar ID stored**: `doctor.calendarId` is populated with primary calendar ID
- □ **Token refresh**: Wait for token to expire (or test manually) → verify automatic refresh
- □ **Permanent failure**: Revoke the app from Google account → verify token marked `revoked`
- □ **Disconnect**: Use admin disconnect → verify token revoked and cleaned up

### 6.3 Push Notifications

- □ **Channel creation**: After connecting a doctor, `CalendarChannel` table has a new entry
- □ **Channel status**: Status is `active`, expiration is 7 days in the future
- □ **Webhook reachable**: `POST /api/google/webhook` returns 200 (test with curl)
- □ **Header validation**: Missing headers return 400
- □ **Channel token validation**: Mismatched token returns 403
- □ **Replay protection**: Same `x-goog-message-number` within 24h is deduplicated
- □ **Sync notification**: State `sync` returns 200 without processing
- □ **Exists notification**: State `exists` triggers `processSync()`

### 6.4 Retry Queue

- □ **Job enqueued**: Failed sync creates a `CalendarSyncJob` entry
- □ **Cron running**: Worker executes every 60 seconds (check logs)
- □ **Lock acquired**: Only one worker runs at a time (409 if locked)
- □ **Stuck recovery**: Jobs stuck in `processing` for >10 min reset to `pending`
- □ **Max attempts**: After 5 attempts, job moves to `failed` permanently
- □ **Escalating delays**: Retry intervals follow `[1, 5, 15, 60, 1440]` minutes
- □ **Admin dashboard**: Retry Queue tab shows pending/failed jobs

### 6.5 Google Meet

- □ **Meet created**: Event created with `conferenceDataVersion: 1`
- □ **Meet link stored**: `booking.calendarLink` contains the `hangoutLink`
- □ **Meet link clickable**: Link opens Google Meet in browser

### 6.6 Watch Channels

- □ **Channel renewal**: Old channel stopped, new channel created before expiry
- □ **24h window**: Renewal triggers when channel expires within 24 hours
- □ **Dedup**: Duplicate channels for same doctor/calendar are stopped
- □ **Cleanup**: Obsolete channels (stopped/error, older than 7 days) are deleted
- □ **Admin dashboard**: Channels tab shows active, expiring, expired channels

### 6.7 Drift Monitor

- □ **Drift check runs**: Trigger via admin dashboard or cron
- □ **Missing events detected**: Google events without matching bookings
- □ **Orphan events detected**: Bookings with event ID but no Google event
- □ **Modified events detected**: Time/duration differences >1 minute
- □ **Status calculated**: `healthy`, `degraded`, or `unhealthy` per doctor
- □ **Audit logged**: `DRIFT_DETECTED` entries in audit log

### 6.8 Scheduling Engine

- □ **Smart availability**: `/api/scheduling/availability` returns slots with buffers
- □ **Multi-doctor**: `/api/scheduling/multi-doctor` returns intersection
- □ **Conflict resolution**: `/api/scheduling/conflict` returns ranked suggestions
- □ **Waiting list**: Add patient → auto-assign on cancellation
- □ **Auto-reschedule**: Doctor unavailable → bookings moved to nearest date/time
- □ **Optimization**: `/api/scheduling/optimize` returns day suggestions
- □ **Forecasts**: `/api/scheduling/forecast` returns predictions
- □ **Analytics**: `/api/scheduling/analytics` returns utilization, trends, heat map

### 6.9 Calendar Admin Dashboard

- □ **Overview tab**: Shows health score, OAuth status, channel status, retry queue
- □ **Metrics tab**: Sync activity bar chart, success/failure ratio
- □ **Activity tab**: Timeline with filters, infinite scroll
- □ **Doctors tab**: Table with per-doctor status and action buttons
- □ **Verification tab**: Booking ID search → 7 check results
- □ **Conflicts tab**: Conflict counts + DataGrid with type filter
- □ **Retry Queue tab**: DataGrid with booking/doctor/attempt/status/error
- □ **Channels tab**: Table with renewal and stop actions
- □ **OAuth tab**: Token status per doctor with reconnect button
